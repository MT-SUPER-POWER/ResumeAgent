# 对于结束 MVP 之后的思考

我觉得 HR 看完简历、发现不错的候选人之后，下一步大概率不是继续“分析”，而是进入 招聘推进流程。
也就是说，前几个页面解决的是：上传简历 → AI 分析 → 查看分析记录 → 找出不错的人

> 那么后面应该接的是：
把不错的人收进短名单 → 横向对比候选人 → 推进面试 → 通知/协作 → 形成结论

> 我们短名单的人之间还得做一次对比，因为简历一筛做的事是把所有不同方向的通通拿来，我们的对比往往是在同方向的人之间还得做一次横向对比

# 人才库

刚才说过，我们初选只能看到这次分析中最高分的人，但是他们追求的岗位是有限的，所以我们的人才库可以把个个方向优秀人的人收集起来，来做横向对比。

然后从这个页面出发，考虑后续的流程

人才库用 JD 来分类，这样子就可以按照不同的岗位来对加入"人才库"的，做一次横向对比

这个库里面的人才表的任意一行，点击可以出现侧边图，然后侧边图里面可以加一个日期 tab，用于做这个人的排期

排期之后的面试希望每一个环节都能够留档，反应面试者在环节中做出的反应（面试官回馈）


## 思考的问题

1. 面试流程可能不是那么固定，不能确保我们的面数，状态轮状和排期要如何处理呢
> 不用担心状态管理，我们呢，我们直接搞一个 [] 数组，里面是随意添加环节
> [笔试、一面、二面、技术面]，随便加，然后完成一个就踢掉最前面的，第一个状态就是人才库的这个人处于阶段

2. 面试 + 面反馈 流程安排

```
人才库选择候选人
  ↓
进入候选人详情
  ↓
排期安排：创建技术一面
  ↓
面试完成后：点击记录反馈
  ↓
填写反馈抽屉：提交归档
  ↓
面试记录页：新增一条技术一面记录
  ↓
点击安排下一轮
  ↓
回到排期安排页创建下一轮
```

## 想流程的时候突然想到的点子


### 人才库横向对比（反馈数据）

因为每一个环节都是留痕了的，那么，可以最后在做一次总结。或者作为几个横向都很厉害的人之间的对比的关键差距

### 面试官的面试经

场景：HR 看完简历之后，可能会想 --> 这个人看起来不错，但我不知道该怎么问。

所以可以提供：

基于简历生成面试问题
基于岗位 JD 生成技术追问
基于风险点生成验证问题
基于项目经历生成深挖问题

例如：

候选人项目中提到了「React 性能优化」。
建议追问：
1. 你具体优化了哪些性能指标？
2. 你如何定位性能瓶颈？
3. 你有没有使用 React.memo、useMemo、虚拟列表？
4. 优化前后的数据对比是什么？
