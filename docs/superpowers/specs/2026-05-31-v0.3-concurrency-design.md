# V0.3 并发处理设计文档

> 版本: V1.0 | 日期: 2026-05-31 | 状态: 设计阶段 | 关联: `docs/architecture/modules.md`, `docs/roadmap.md`

## 一、目标

将当前串行 pipeline 改造为 Producer-Consumer 并发架构，支持多份简历并行处理，提升百份级别简历的吞吐量。

### 1.1 验收标准

- 100 份简历，concurrency=5 时，总耗时 < 15 分钟
- 不触发 API rate limit 错误
- 单份简历任一阶段失败不影响其余
- 汇总报告包含失败清单

---

## 二、架构

### 2.1 Producer-Consumer 管道

> 详细设计见 **[Pipeline 并发模型](../../architecture/pipeline-concurrency.md)**，以下为要点摘要。

**架构**：3-stage Producer-Consumer，Work-stealing 任务分配。

- **Stage 1（M=2 parse workers）** — Work-stealing 分配文件，Semaphore(M) 控制并发解析，产出推入 bounded channel
- **Stage 2（N evaluate workers）** — 从 channel 争抢 ParseTask，Semaphore(N) 控制并发 LLM 调用，结果落库
- **Stage 3（1 consumer）** — 等待全部完成，DB JOIN pipeline_states 过滤本 run 结果，生成报告 + 失败清单

**任务分配**：`Arc<AtomicUsize>` + `fetch_add(1)`，不预分组，快的 worker 多抢多干。

### 2.2 并发控制

```
M = pipeline.parse_workers     # Stage 1 解析并发，默认 2
N = llm.concurrency.value()    # Stage 2 评估并发，auto=3

concurrency 来源（优先级从高到低）:
  1. CLI: --concurrency 5       → 覆盖 N
  2. application.yaml           → N 和 M 分别配置
  3. auto 默认值                → M=2, N=3
```

### 2.3 数据库连接池

```
needed = (M + N) × 2 + 2
if needed > db.max_connections:
    warn 建议调大
继续执行（不阻塞）
```

### 2.4 汇总汇聚

Stage 3 不直接查 evaluations 表，而是 JOIN pipeline_states 做 run-scope 过滤：

```sql
SELECT e.* FROM evaluations e
JOIN pipeline_states ps ON ps.resume_id = e.resume_id
WHERE ps.job_run_id = $run_id
  AND ps.phase = 'evaluate'
  AND ps.status = 'success'
ORDER BY e.talent_score DESC
```

失败清单来源：

```sql
SELECT resume_id, phase, error_msg FROM pipeline_states
WHERE job_run_id = $run_id AND status = 'failed'
```

## 三、配置

### 3.1 application.yaml

```yaml
pipeline:
  parse_workers: 2            # M — Stage 1 解析并发数

llm:
  provider: openai
  concurrency: auto           # N — Stage 2 评估并发数，auto=3
  max_retries: 3
  # ...

db:
  max_connections: 20
```

### 3.2 CLI

```bash
resume-agent run --dir ./resumes --concurrency 5    # 手动覆盖
resume-agent run --dir ./resumes                    # 读配置文件
resume-agent rerun --concurrency 5                  # rerun 也可覆盖
```

### 4.3 数据结构（Rust）

```rust
struct FailureEntry {
    file_name: String,
    phase: String,       // "parse" | "evaluate"
    reason: String,
}

struct RunSummary {
    total: i32,
    parsed: i32,
    evaluated: i32,
    failed: i32,
}

struct SummaryJson {
    title: String,
    generated_at: String,
    job_run_id: String,              // 新增
    summary: RunSummary,             // 新增
    stats: serde_json::Value,
    rankings: Vec<RankingEntry>,
    failures: Vec<FailureEntry>,     // 新增
}
```
