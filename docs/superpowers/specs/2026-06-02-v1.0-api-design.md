# V1.0 REST API 设计

> 状态: 设计完成
> 日期: 2026-06-02
> 关联: `docs/roadmap.md` V1.0 章节, `repo/backend/docs/reference/api-response-format.md`

---

## 概述

ResumeAgent V1.0 聚焦 REST API 服务化：

- **V1.0（本文档）**: 所有 CLI 命令抽象为 `/api/v1/*` REST API，Auth 机制，SSE 实时推送
- **V1.1（后续）**: 基于 Next.js + shadcn 的前端管理面板（Admin + HR 双角色）

### 关键设计决策

| 决策 | 结论 |
|---|---|
| Web 框架 | Actix-web，嵌入现有 Rust 项目，`resume-agent serve --port 8080` 启动 |
| 长任务模式 | 异步提交 + SSE 进度推送 + `GET /jobs/{id}` 权威状态 |
| 认证 | users 表 + bcrypt 密码 + JWT（Rust 侧签发 + 校验），种子 admin/admin123 |
| 角色 | `admin` / `user`，JWT payload 携带 role，用户管理 admin only |
| 文件提交 | multipart/form-data 批量上传，分析完成后自动清理临时文件 |
| 结果格式 | 纯结构化 JSON，Markdown 渲染交给前端 |
| 分页 | query string `?page=1&size=20` |

## 2. 统一响应信封

```
成功: { "code": 0, "data": { ... }, "msg": "ok" }
失败: { "code": 40001, "msg": "参数校验失败", "data": null, "detail": { ... } }
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `code` | `u16` | `0` = 成功，非 `0` = 业务错误码 |
| `msg` | `String` | 人类可读消息 |
| `data` | `T \| null` | 成功时携带数据体，失败时为 `null` |
| `detail` | `Value \| null` | 失败时的可选结构化详情（字段校验错误等） |

### HTTP 状态码

- `200` — 所有业务响应（包括业务错误），从 `code` 区分成败
- `500` — 仅服务端未知异常

### 分页

```json
// GET /api/v1/jobs?page=1&size=20
{
  "code": 0,
  "msg": "ok",
  "data": {
    "list": [ ... ],
    "page": 1,
    "page_size": 20,
    "total": 47
  }
}
```

## 3. 错误码

```
0       — 成功

10xx    — 参数校验
  10001   缺少必填字段
  10002   字段值不合法（如 protocol 不是 openai/claude）
  10003   文件格式不支持（非 PDF/DOCX）
  10004   文件为空或损坏

20xx    — 资源
  20001   Job 不存在
  20002   JD 不存在
  20003   Model 不存在
  20004   Resume 不属此 Job

30xx    — 业务逻辑
  30001   任务正在运行中，不允许重复 rerun
  30002   无可用模型（所有模型 disabled 或未注册）
  30003   上传文件列表为空
  30101   用户名已存在
  30102   用户名或密码错误
  30103   账户已禁用（is_active = false）
  30104   不能删除自己

40xx    — 服务端
  40001   DB 连接失败
  40002   LLM 调用失败（所有重试耗尽）
  40003   文件系统 IO 错误
  40301   权限不足（非 admin 访问管理端点）

50xx    — 管理
  50001   模型名重复
  50002   API Key 加密失败
  50003   无法删除默认模型（需先设另一个默认）
  50004   协议不支持
```

## 4. 认证架构

### users 表

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | UUID PK | |
| `username` | String UNIQUE | |
| `password_hash` | String | bcrypt |
| `role` | `"admin"` \| `"user"` | |
| `is_active` | Boolean | 默认 true，软删除标记 |
| `created_at` | Timestamp | |
| `updated_at` | Timestamp | |

### 种子数据

```sql
INSERT INTO users (id, username, password_hash, role, is_active)
VALUES (gen_random_uuid(), 'admin', bcrypt('admin123'), 'admin', true);
```

### JWT 流程

```
POST /api/v1/auth/register  { "username": "zhang", "password": "123456" }
  → username 唯一性检查
  → bcrypt 哈希 → INSERT users (role='user', is_active=true)
  → jwt.sign({ sub: uuid, username, role }, SECRET, 24h)
  → 返回 { token: "eyJ..." }

POST /api/v1/auth/login  { "username": "admin", "password": "admin123" }
  → SELECT * FROM users WHERE username = 'admin' AND is_active = true
  → bcrypt 验证密码
  → jwt.sign(...)
  → 返回 { token: "eyJ..." }

后续所有请求:
  Authorization: Bearer eyJ...
  → JWT middleware 验证签名
  → 注入 request extensions: { sub, username, role }
  → 管理端点 handler 层检查 role == "admin"
```

### JWT Payload

```json
{ "sub": "uuid", "username": "admin", "role": "admin", "exp": 1717400000 }
```

### 暂不做的

- 密码复杂度限制
- 邮箱验证 / magic link
- 登录失败次数限制
- 修改密码接口
- refresh token（24h 过期后重新登录）

## 5. SSE 事件规范

### 设计原则

SSE 推送的是**状态快照**，不是事件日志。前端收到后直接覆盖 UI 状态，无需自行推演状态机。
`GET /api/v1/jobs/{id}` 是权威状态来源，SSE 只是实时广播。断线重连后前端必须用 GET 拉当前状态。

### 统一基础字段

每条事件都带：

```json
{ "job_id": "...", "stage": "...", "timestamp": "..." }
```

`stage` 默认继承上一条 `stage_started` 的值。

### 事件清单

| 事件 | 触发时机 | 关键字段 |
|---|---|---|
| `job_started` | 异步 Worker 真正启动 | `{status, total}` |
| `stage` | 阶段切换 | `{stage, status: "started"\|"completed", total}` |
| `progress` | 单文件完成 + job 全局快照 | `{done, total, file, resume_id, message, parsed, evaluated, failed}` |
| `warning` | 非致命告警（skipped、无文本层等） | `{file, resume_id, code, message}` |
| `error` | 单文件评估失败 | `{file, resume_id, code, message, attempt}` |
| `heartbeat` | 每 15s | `{}` |
| `complete` | 任务终态 | `{status, summary: {total, evaluated, failed, ...}}` |

### progress payload 完整结构

```json
{
  "job_id": "abc123",
  "timestamp": "2026-06-02T10:30:05Z",
  "stage": "evaluating",
  "file": "张三简历.pdf",
  "resume_id": "res_3f8a",
  "message": "评估完成 (claude-sonnet-4-6)",
  "done": 7,
  "total": 20,
  "attempt": 1,
  "parsed": 20,
  "evaluated": 7,
  "failed": 1
}
```

### summarizing 阶段

summarizing 走阶段事件，不做逐文件进度：

```
event: stage
data: {"job_id": "...", "stage": "summarizing", "status": "started", "total": 20}

event: stage
data: {"job_id": "...", "stage": "summarizing", "status": "completed"}

event: complete
data: {"job_id": "...", "status": "completed", "summary": {"total": 20, "evaluated": 19, "failed": 1, ...}}
```

## 6. 后端分层架构

### 模块布局

```
src/
├── main.rs              # CLI subcommand + serve 入口
├── cli/                 # 现有 CLI 命令（不动）
├── api/                 # ★ 新增
│   ├── mod.rs           #    路由注册，actix HttpServer 启动
│   ├── middleware.rs    #    JWT 验证 + Role 注入
│   ├── auth.rs          #    register/login/me 业务逻辑
│   ├── handlers/
│   │   ├── mod.rs
│   │   ├── job.rs       #    POST/GET jobs, rerun, results
│   │   ├── jd.rs        #    JD CRUD
│   │   ├── model.rs     #    model 只读 + 管理
│   │   ├── user.rs      #    用户管理（admin only）
│   │   └── system.rs    #    DB status, reset, cache clean
│   └── sse.rs           #    SSE 连接管理 + 事件广播
├── config/              # 现有（不动）
├── database/            # 现有（不动）
├── errors/              # 现有 + ApiResponse + AppError
├── llm/                 # 现有（不动）
├── models/              # 现有（不动）
├── pipeline.rs          # 现有，加 SSE 回调钩子
├── repository/          # 现有（不动）
├── services/            # 现有（不动）
└── utils/               # 现有（不动）
```

### 调用链

```
HTTP Request
  → JWT middleware (验证 token，注入 role)
  → Handler (参数校验、组装)
  → Service / Repository (复用现有)
  → Pipeline (复用现有，注入 SSE sender)
  → ApiResponse::ok(...)
```

### SSE 广播机制

```
pipeline.rs
  │  持有 tokio::broadcast::Sender
  │  每完成一个阶段/文件 → send(event)
  ▼
sse.rs
  │  GET /jobs/{id}/sse 订阅 broadcast::Receiver
  │  转换为 SSE 字节流 (text/event-stream)
  ▼
前端 EventSource
```

每个 job 一个独立的 `broadcast::channel`。Job 完成后 channel 保留 5 分钟（允许前端重连补数据），然后释放。心跳事件保持连接活跃。

### Pipeline 集成点

```rust
// pipeline.rs 现有签名
pub async fn run(files: &[PathBuf], model: &str, concurrency: Option<u32>, options: RunOptions) -> Result<()>

// 新增可选参数
pub async fn run(
    files: &[PathBuf],
    model: &str,
    concurrency: Option<u32>,
    options: RunOptions,
    sse_tx: Option<tokio::sync::broadcast::Sender<SseEvent>>,  // ★ 新增
) -> Result<()>
```

`None` = CLI 模式（现有行为），`Some(tx)` = API 模式（推送事件）。Pipeline 核心业务逻辑不变。

## 7. 文件生命周期

```
POST /api/v1/jobs (multipart)
  → 文件存入 /tmp/resume-agent/uploads/{job_id}/
  → Pipeline 解析阶段读文件 → SHA256 去重 → 插入 DB
  → Pipeline 评估阶段从 DB 读解析结果（不再读文件）
  → Job 完成后: 删除 /tmp/resume-agent/uploads/{job_id}/
  → 手动清理: DELETE /api/v1/system/cache
```

恢复已存在 SHA256 的文件自动跳过上传，复用已有解析结果。

## 8. 删除策略

所有删除均为软删除，保留历史引用完整性：

| 实体 | 机制 | 说明 |
|---|---|---|
| JD | `is_active = false` | V0.1 已有字段 |
| User | `is_active = false` | 禁用登录，保留 job_run.user_id 引用 |
| Model | `enabled = false` | V0.4 已有字段，disable 后不可选用 |
| Job | 不可删除 | 无删除端点，保留完整评估历史 |

## 9. 技术债

以下明确不在 Phase 1 范围内，留给后续版本：

| # | 项 | 原因 |
|---|---|---|
| 1 | 用户工作空间隔离 | 无 workspace 概念，模型和 job 全局可见 |
| 2 | 模型分层（官方 vs 自定义） | sync-prices 行为与角色权限交叉，需先设计分层模型 |
| 3 | 密码安全增强 | 复杂度限制、邮箱验证/magic link、登录失败次数限制、修改密码 |
| 4 | Token 撤销机制 | 无黑名单、无 refresh token，过期后重新登录 |
| 5 | 模型价格历史审计 | 调价后历史 job_run 回溯当时单价 |
| 6 | Rate Limiting | V0.2 重试已覆盖基本韧性，API 层限流后续加 |
