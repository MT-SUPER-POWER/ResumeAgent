# V1.2 人才库面试闭环设计

> 状态: Draft
> 创建: 2026-06-05
> 关联: `docs/roadmap.md` V1.2, `docs/ui-design/design.md` 人才库章节

## 1. 背景与目标

### 1.1 问题

V1.1A 完成了前端底座和人才库 UI（mock 数据），但人才库处于"只读"状态：

- 候选人可以从分析结果查看，但无法正式归档到人才库。
- 排期安排和面试记录 Tab 的 UI 已有，但数据不持久化。
- 面试反馈面板只是静态展示，没有提交和归档能力。
- 候选人阶段无法推进——一条招聘流程进去就出不来。

**一句话**：人才库是一个"展示柜"，不是"工作台"。

### 1.2 目标

打通一条完整的招聘闭环电路：

```
人才库筛选 → 安排面试 → 会议记录留痕 → 阶段同步回表
   ↑                                                      |
   └──────────────────── 闭环 ────────────────────────────┘
```

具体：

1. HR 从分析结果中把候选人加入人才库，按 JD 归档。
2. HR 在人才库中为候选人安排面试（时间、面试官、方式）。
3. 面试完成后，HR 通过"记录反馈"面板提交面试结论、评分维度、亮点/疑点/风险/建议追问。
4. 提交反馈后，候选人阶段自动推进（`stageQueue` 队首出队）。
5. 面试记录归档到时间线，不可编辑但可查看。

### 1.3 数据流通全景

闭环的核心是数据在四张表之间按确定的规则流动。以下用一次完整的招聘流程说明数据如何在各环节之间传递和变换。

```
                              ▼
```

```
│                                                                    │
│ HR 继续安排面试 → STEP 2 → STEP 3 → STEP 4 → ...                   │
│                                                                    │
│ 直到 stage_queue 为空:                                             │
│   talent_pool.communication_status = "已归档"                      │

│                                                                    │
│ 面试记录 Tab 时间线积累完整历史:                                    │
│   ┌─ HR 初沟通     已完成  ─┐                                      │
│   ├─ 技术一面       已完成  ─┤  ← 不可编辑但可查看                   │
│   ├─ 作业评估       已完成  ─┤                                      │
│   └─ 复试          已完成  ─┘                                      │
└────────────────────────────────────────────────────────────────────┘
```

#### 1.3.3 数据一致性的关键保证

```
                    ┌─────────────────────────────────┐
                    │     数据一致性约束                 │
                    ├─────────────────────────────────┤
                    │                                 │
                    │  1. 唯一性约束                    │
                    │     talent_pool:                 │
                    │     UNIQUE(resume_id, jd_id)     │
                    │     → 同一候选人同一JD不会重复添加  │
                    │                                 │
                    │  2. 外键约束                      │
                    │     talent_pool.resume_id        │
                    │       → resumes(id) 必须存在      │
                    │     talent_pool.jd_id            │
                    │       → job_descriptions(id)     │
                    │     interviews.talent_pool_id    │
                    │       → talent_pool(id) 必须存在  │
                    │                                 │
                    │  3. 阶段推进原子性                 │
                    │     advance_stage() 使用           │
                    │     SELECT ... FOR UPDATE        │
                    │     防止并发操作导致阶段跳跃        │
                    │                                 │
                    │  4. 状态流转约束                   │
                    │     interviews.status:            │
                    │       已安排 → 已完成 → (已取消)   │
                    │     talent_pool.communication_    │
                    │     status:                       │
                    │       待沟通 → 已确认 → 待反馈     │
                    │                     → 已归档      │
                    │                                 │
                    │  5. 反馈不可变性                   │
                    │     已完成的 interview 的反馈字段  │
                    │     仅允许一次写入，不支持覆盖修改  │
                    │     (可在 API 层校验 status)       │
                    └─────────────────────────────────┘
```

#### 1.3.4 数据流通中的关键 ID 追踪

每条数据在闭环中的可追溯性：

```
候选人 "林晓棠" 在系统中的完整数据链:

evaluations
  ├─ id=42, resume_id=101, matched_jd_id="jd-ai"
  │   talent_score=86.4, match_score=88.7
  │
  └──→ talent_pool
         ├─ id=1, resume_id=101, jd_id="jd-ai"
         │   source_job_id="A719"        ← 可追溯到哪次分析
         │   source_batch="批次 #A719"    ← 可追溯到哪个批次
         │
         ├──→ interviews id=1  (HR初沟通, 已完成)
         ├──→ interviews id=2  (技术一面, 已完成)
         │      overall_score=85.0
         │      conclusion="进入下一轮"
         ├──→ interviews id=3  (作业评估, 已完成)
         └──→ interviews id=4  (复试, 已安排)

任何一条 interview 都可以通过 talent_pool_id → resume_id → evaluations 找到:
  - 原始简历文件
  - AI 评分的 talent_score / match_score
  - 完整的 result_json
  - 来源分析任务和批次
```

## 2. 数据模型

### 2.1 ER 关系

```
evaluations ──────────────┐
  (已有)                   │ 1:1 (resume_id)
                           ▼
                      talent_pool ──────┐
                        │                │ 1:N
                        │ job_descriptions (已有)
                        │
                        │ 1:N
                        ▼
                     interviews
```

- `talent_pool` 是跨 Job 的人才库主表，一个候选人（resume）在同一个 JD 下只有一条人才库记录。
- `interviews` 是面试记录，一个候选人可以有多轮面试。

### 2.2 talent_pool 表

```sql
CREATE TABLE IF NOT EXISTS talent_pool (
    id                  SERIAL PRIMARY KEY,
    resume_id           INT NOT NULL REFERENCES resumes(id) ON DELETE CASCADE,
    jd_id               TEXT NOT NULL REFERENCES job_descriptions(id),
    source_job_id       TEXT,                          -- 来源分析任务 ID
    source_batch        TEXT,                          -- 来源批次名
    owner               TEXT NOT NULL DEFAULT '',      -- 负责人
    communication_status TEXT NOT NULL DEFAULT '待沟通', -- 待沟通 / 已确认 / 待反馈 / 已归档
    stage_queue         TEXT[] NOT NULL DEFAULT '{}',  -- 阶段队列，队首 = 当前阶段
    added_at            TIMESTAMP(0) DEFAULT now(),
    updated_at          TIMESTAMP(0) DEFAULT now(),

    UNIQUE(resume_id, jd_id)  -- 同一候选人同一 JD 只有一条记录
);

CREATE INDEX idx_talent_pool_jd ON talent_pool(jd_id);
CREATE INDEX idx_talent_pool_owner ON talent_pool(owner);
CREATE INDEX idx_talent_pool_status ON talent_pool(communication_status);
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `resume_id` | INT FK | 关联 resumes 表 |
| `jd_id` | TEXT FK | 关联 job_descriptions，作为人才库分类依据 |
| `source_job_id` | TEXT | 来源分析任务 ID，方便追溯到原始分析 |
| `source_batch` | TEXT | 来源批次名 |
| `owner` | TEXT | 负责人（HR 姓名或 ID） |
| `communication_status` | TEXT | 沟通状态枚举 |
| `stage_queue` | TEXT[] | 阶段队列，如 `{"待沟通","技术一面","作业评估","复试"}` |

### 2.3 interviews 表

```sql
CREATE TABLE IF NOT EXISTS interviews (
    id                  SERIAL PRIMARY KEY,
    talent_pool_id      INT NOT NULL REFERENCES talent_pool(id) ON DELETE CASCADE,
    round               TEXT NOT NULL,                 -- 轮次名称，如 "技术一面"
    scheduled_at        TIMESTAMP(0),                  -- 排期时间
    duration_minutes    INT DEFAULT 60,                -- 预计时长
    status              TEXT NOT NULL DEFAULT '已安排', -- 已安排 / 已完成 / 已取消
    interviewers        TEXT[],                        -- 面试官列表
    location            TEXT,                          -- 线上会议 / 线下地址
    -- 以下为面试反馈字段（status='已完成'后填写）
    conclusion          TEXT,                          -- 通过 / 待定 / 淘汰 / 进入下一轮
    overall_score       DOUBLE PRECISION,             -- 综合评分 (0-100)
    score_dimensions    JSONB,                        -- [{label, score, maxScore}] 评分维度
    highlights          TEXT[],                        -- 亮点
    concerns            TEXT[],                        -- 疑点
    risks               TEXT[],                        -- 风险点
    suggested_questions TEXT[],                        -- 建议追问
    notes               TEXT,                          -- 面试官原始备注
    attachments         JSONB,                        -- [{name, size, type, url}] 附件列表
    created_at          TIMESTAMP(0) DEFAULT now(),
    updated_at          TIMESTAMP(0) DEFAULT now()
);

CREATE INDEX idx_interviews_pool ON interviews(talent_pool_id);
CREATE INDEX idx_interviews_status ON interviews(status);
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `talent_pool_id` | INT FK | 关联 talent_pool |
| `round` | TEXT | 轮次名称，不强制枚举 |
| `scheduled_at` | TIMESTAMP | 排期时间 |
| `status` | TEXT | 已安排 → 已完成 → (可选)已取消 |
| `interviewers` | TEXT[] | 面试官姓名列表 |
| `conclusion` | TEXT | 面试结论，用于推进或终止流程 |
| `score_dimensions` | JSONB | 结构化评分维度 |
| `highlights/concerns/risks/suggested_questions` | TEXT[] | 面试反馈四要素 |
| `attachments` | JSONB | 面试纪要/作业文件等附件 |

### 3.2 面试管理

#### `POST /api/v1/talent-pool/{id}/interviews`

安排一轮面试。

**Request Body:**

```json
{
  "round": "技术一面",
  "scheduledAt": "2026-06-17T15:00:00Z",
  "durationMinutes": 60,
  "interviewers": ["张工", "李工"],
  "location": "线上会议"

```

**Response `data`:** 新创建的 interview 记录。

#### `GET /api/v1/talent-pool/{id}/interviews`

获取某候选人的所有面试记录，按时间倒序。

#### `PATCH /api/v1/interviews/{id}`

更新面试记录（面试完成后填写反馈）。

**Request Body (partial update):**

```json
{
  "status": "已完成",
  "conclusion": "进入下一轮",
  "overallScore": 85.0,

  ],

  "risks": ["高并发经验有限"],
  "suggestedQuestions": ["请说明..."],
  "notes": "整体表现良好"
}
```

#### `POST /api/v1/talent-pool/{id}/advance-stage`

推进阶段：队首出队，下一阶段自动成为当前阶段。

**Request Body (可选):**

```json
{
  "autoCreateInterview": true,
  "nextRound": "复试",
  "scheduledAt": "2026-06-24T14:00:00Z"
}
```

**行为:**
1. `stageQueue` 队首出队。
2. 如果 `autoCreateInterview=true`，自动创建下一轮面试记录。
3. 如果队列为空，`communicationStatus` 标记为 `已归档`。

## 6. 与现有系统的集成点

### 6.1 从分析结果添加到人才库

在 `GET /api/v1/jobs/{id}/results` 的响应中，每条结果增加 `inTalentPool: boolean` 字段，前端据此显示"已加入人才库"或"添加到人才库"按钮。

### 6.2 人才库与 evaluations 的关系

- 人才库不复制 evaluation 数据，而是通过 `resume_id` JOIN evaluations 获取评分信息。
- 如果同一候选人有多条 evaluation（多次分析），展示最新的一条。
- 人才库的 `jd_id` 与 evaluation 的 `matched_jd_id` 可以不同——HR 可以手动指定 JD 分类。

### 6.3 前端路由

不新增路由。人才库所有交互在 `/dashboard/candidates` 页面内通过 Sheet/Dialog 完成：

- 详情查看 → `CandidateDetailSheet`
- 面试反馈 → `InterviewFeedbackPanel`（二级覆盖面板）
- 从分析记录添加 → `/dashboard/jobs/[id]` 页面内的按钮

## 8. 风险与注意事项

| 风险 | 缓解 |
|------|------|
| 面试反馈数据结构复杂（JSONB 嵌套） | Rust handler 层做 schema 校验，前端表单做 zod 校验 |
| stageQueue 操作并发冲突 | 使用 PostgreSQL 行锁（`SELECT ... FOR UPDATE`）保证原子推进 |
| 前端 mock 数据与真实 API 结构差异 | 先对齐类型定义，再逐步替换数据源 |
| 附件上传（面试纪要文件） | V1.2 先用简单的 multipart upload + 本地存储，不做 CDN |
